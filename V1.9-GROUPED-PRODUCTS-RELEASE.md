# Pool Shed v1.9 — Grouped Products and Variants

## Architecture
Existing `products.id` records remain the operational stock-controlled variant IDs. A new `product_parents` table stores shared catalogue information. Existing products are backfilled to one parent plus one default variant, so PO, SO, stock, allocation, goods-in, movement and return foreign keys remain valid.

## Affected files
- `public/index.html` — grouped Product Hub, responsive variant tables/cards, grouped search, variant-aware selectors, grouped import preview and validation.
- `worker/index.js` — aligned application source.
- `migrations-v1.9-grouped-products.sql` — additive Supabase migration, indexes, constraints, audit table, reporting view and rollback notes.
- `templates/the-pool-shed-product-bulk-template-v1.9.csv` — grouped import template and acceptance-test catalogue.
- `public/service-worker.js` — release cache increment.

## Deployment order
1. Back up production Supabase.
2. Run the migration on staging and verify row counts.
3. Deploy this package to a Vercel preview connected to staging.
4. Import the supplied CSV and complete the acceptance checks.
5. Run the migration on production.
6. Merge the tested code to `main`.

## Rollback
Redeploy the previous Vercel build. The additive parent/variant tables can safely remain unused. A destructive schema rollback is documented at the bottom of the migration and should only be used after a verified backup.
